## 專案概述

在寶馬，我們致力於確保每一輛新車都能在全國各地暢行無阻，享受便捷的充電體驗。本專案的核心任務是為新車的全國路測設計一套科學、高效的充電路線規劃解決方案。

我們的目標是：**在滿足各種複雜限制條件（如時間、預算、車輛性能等）的前提下，制定出最少的測試路徑，以實現最大範圍的充電網絡覆蓋率。**

這個工具將為我們的路測團隊提供強大的模擬和規劃能力。透過輸入目標城市、出發酒店、測試車輛參數以及待測充電站列表，系統能夠模擬並生成最佳化的每日測試路線，以最高效率完成所有測試任務。

## 核心概念

项目中主要处理两种类型的充电站数据：

1. **CPO (Charge Point Operator) 数据**:

   - 来源：我们自有或主要的充电运营商数据集 (`national_charge_station.csv`)。
   - 特点：数据结构统一，覆盖范围广，是我们网络中的基本盘。
2. **友商/OEM (Original Equipment Manufacturer) 数据**:

   - 来源：通常需要针对特定城市，通过地图服务等第三方渠道**额外抓取**（例如 `guangzhou_oem_stations.csv`）。
   - 特指：竞争对手品牌（如比亚迪、小米、蔚来、理想等）自建或合作的充电站。
   - 重要性：这些是路测任务中需要被重点关注和评估的目标。

## 核心选站策略

在模拟开始前，系统会通过特定策略从成千上万的充电站中筛选出一个精简、高效的待测列表。

### 策略 A: 功率优先 (Power Priority)

- **目标**: 覆盖尽可能多的 **“品牌-功率类型”** 组合。
- **逻辑**: 为每种技术组合（如“星星充电-超充”）选择一个离酒店最近的站点作为代表。
- **适用场景**: 当测试目标是全面摸底不同运营商的技术能力时。

### 策略 D: 统一逻辑 (Unified Strategy)

- **目标**: **强制覆盖重点友商 (OEM) 站点**，同时兼顾 CPO 网络的广度。
- **逻辑**:
  1. **强制选择**: 优先从友商 (OEM) 数据中，根据评分选出高、低分区的代表站。
  2. **补充缺口**: 检查这些友商代表覆盖了哪些技术组合，然后从 CPO 数据池中选择站点来补全剩余的技术组合。
- **适用场景**: 这是项目当前**最核心、最全面**的策略，因为它确保了重点的友商站点一定会被测试。

## 如何扩展到新城市

要将此模拟器应用于新的城市，最关键的一步是准备数据：

1. **准备友商数据**: 必须提供一个针对新城市的友商充电站数据文件，格式应为 `[城市名]_oem_stations.csv`。该文件需要包含至少 `name`, `rating`, `brand_keyword`, `longitude`, `latitude` 等关键字段。
2. **更新酒店数据**: 确保 `datasets/hotels_with_address.csv` 文件中包含新城市的目标酒店信息，或在代码中动态添加。
3. **路网数据缓存**: 首次在新城市运行时，系统会通过高德地图 API 获取路网信息并生成 `amap_route_cache.csv`。这会花费较长时间。后续运行则会直接使用缓存，速度会极大提升。

## 照片管理與 station_id 規範（概要，2026-02-28）

以下為專案近期討論並採納的照片管理與命名規範，便於展示頁面和後續自動化處理：

- 目錄結構（推薦）:
  - `images/structured/{station_id}/` — 每個站點的標準化資料夾，包含四張標準照片與 `meta.json`。
    - `pile_left.jpg`, `pile_right.jpg`, `gun.jpg`, `plate.jpg`, `others/`, `meta.json`
  - `images/raw/` — （可選）原始上傳備份，不直接用於展示。
  - `images/index.csv` — 全局索引，快速查詢站點與照片可用性。

- `station_id` 命名建議:
  - 使用 ASCII-safe 的格式：`{CITYCODE}_{CPOCODE}_{SEQ}`（例如 `GZ_TLD_001` 或 `gz_tld_001`）。
  - CityCode：2–3 字母（GZ、SH、CD）；CPOCode：3–6 字母；序號至少 3 位零填充。
  - `station_id` 為系統唯一鍵（canonical key），所有 UI、索引、腳本均以此為基準。

- `meta.json`（每站）— 最小欄位建議：
  - `station_id`, `station_name`, `city`, `cpo`, `latitude`, `longitude`, `address`
  - `photos`: mapping（如 `{"pile_left":"pile_left.jpg", ...}`）
  - `last_updated`（ISO8601），可選 `inspected_by`, `notes`, `tags`

- 全局索引（`images/index.csv`）建議欄位：
  - `station_id,city,station_name,cpo,folder_path,has_left,has_right,has_gun,has_plate,last_updated,notes`

- 簡要工作流程：
  1. 采集者上傳原始照片至 `images/raw/`（或透過上傳介面）。
  2. 執行整理/歸檔腳本：匹配 `station_id`、標準化檔名、移動到 `images/structured/{station_id}/`、產生/更新 `meta.json` 並更新 `images/index.csv`。
  3. 前端（Streamlit）讀取 `images/index.csv` 與 `meta.json`，展示四張標準圖與缺圖佔位。

備註：
- 在顯示層使用中文 `station_name`、`cpo` 作為展示資訊，內部檔案與路徑使用 `station_id`（ASCII）保持一致。
- 建議在 ingestion 階段生成縮圖 `thumbs/` 以節省帶寬。
- 若未來擴展至大量圖片，建議採用物件存儲（S3）並在 `meta.json` 中儲存 URL。

為了讓路線規劃更具體、更貼近真實業務場景，我們定義了以下關鍵目標與限制條件：

### 主要目標 (Objective)

- **充電運營商覆蓋率 (CPO Coverage)**: 測試路線需覆蓋全國 **90%** 以上的充電運營商品牌 (CPO)。

### 限制條件 (Constraints)

- **人員限制**:
  - **每日工時**: 司機每日工作時長不得超過 8 小時。
  - **連續出差**: 連續出差天數不能超過 10 天，之後必須安排休息日。
- **任務限制**:
  - **特定站點**: 必須訪問指定的重點充電站品牌 (OEM) 和在特定時間內需要訪問的站點。
- **預算與法規**:
  - **每日預算**: 需遵守每日或整個任務週期的預算限制。
  - **地方法規**: 必須遵守測試地點當地的交通及其他現行法規。
